# ADR-003: Transactional outbox и NATS JetStream

- Статус: принято
- Дата: 2026-07-12

## Контекст

Запись доменных данных и публикация сообщения в брокер не могут быть одной распределённой транзакцией. Публикация после commit теряет событие при падении процесса, а публикация до commit позволяет потребителю увидеть изменение, которое затем откатилось.

## Решение

Каждый сервис-издатель записывает доменное изменение и строку `outbox` одной транзакцией PostgreSQL. Отдельный relay читает неотправленные строки с блокировкой, публикует их через подтверждаемую JetStream publish и только затем отмечает отправленными. `eventId` передаётся как `Nats-Msg-Id` для дедупликации публикации.

Subject имеет форму `teamos.<service>.<entity>.<action>.v1`. Общий envelope:

```json
{
  "eventId": "<uuid>",
  "occurredAt": "<ISO-8601 UTC>",
  "companyId": "<uuid>",
  "actorId": "<uuid-or-empty-for-system>",
  "payload": {}
}
```

Trace context передаётся заголовками NATS. Потребители durable и работают общей queue group для горизонтального масштабирования. Обработчик атомарно проверяет `processed_events`, применяет изменение и фиксирует `eventId` в одной локальной транзакции; дубликат подтверждается без повторного эффекта. Ошибка приводит к NAK и повтору. После `MaxDeliver` исходное сообщение публикуется в `teamos.dlq.*`, затем подтверждается; DLQ > 0 создаёт алерт.

## Последствия

- Доставка как минимум однократная, а наблюдаемый доменный эффект — однократный при корректном idempotent handler.
- Relay и очистка outbox становятся обязательной частью каждого сервиса-издателя.
- Порядок гарантируется только в пределах выбранного stream/consumer; домены не должны полагаться на глобальный порядок.
- Недоступность NATS не откатывает принятую бизнес-операцию: событие остаётся в outbox и будет опубликовано позже.
