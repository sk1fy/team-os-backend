# ADR-007: Enrollment как единственный источник состояния обучения

- Статус: принято
- Дата: 2026-07-22

## Контекст

Legacy progress привязан к паре `(user_id, course_id)`. Такая модель требует полноценного пользователя TeamOS, не поддерживает независимые повторные прохождения, не различает назначение, персональную ссылку и кампанию и не фиксирует версию курса. Если frontend самостоятельно вычисляет продолжение обучения из нескольких summary-полей, состояние расходится при конкурентных запросах.

## Решение

Каждое конкретное прохождение представлено `CourseEnrollment`. Enrollment обязательно фиксирует компанию, курс, опубликованную версию, ровно одну learner identity и источник доступа.

Learner имеет тип `user` или `external`. Для него заполнен ровно один из `user_id` и `external_learner_id`. Источник различает assignment, personal access, partner promo campaign, company candidate campaign, repeat training и legacy migration. Повторное обучение создаёт новый enrollment с отдельным `attempt_number`, а не очищает старый progress.

Состояние разделено на две оси:

- `progress_status`: `not_started`, `in_progress`, `completed`;
- `access_status`: `invited`, `ready`, `active`, `expired`, `frozen`, `suspended`, `revoked`, `closed`.

Прогресс уроков хранится по `(enrollment_id, lesson_version_id)`. Попытка теста также принадлежит enrollment и versioned quiz. Процент вычисляется по обязательным урокам зафиксированной версии. Текущий урок и resume определяет сервер.

Команда отправки теста атомарно блокирует enrollment, проверяет доступ и лимит попыток, оценивает закрытые вопросы, создаёт attempt, завершает урок, обновляет общий progress, открывает следующий урок, при необходимости завершает курс и записывает outbox. Частично применённое продвижение недопустимо.

Внутренний due date остаётся сроком выполнения для overdue/report и не блокирует контент. Внешний `access_until` является жёсткой границей доступа и проверяется синхронно на каждом read/write path; фоновый expiry worker только материализует уже наступившее состояние.

Legacy DTO во время перехода строится из enrollment model. После cutover legacy summary не является вторым источником истины.

## Последствия

- Одинаковый человек может иметь несколько независимых прохождений одного курса из разных источников.
- Новая версия курса не меняет существующий enrollment.
- Assignment фиксирует версию; повторная аттестация создаёт новое прохождение.
- Все player, report и resume endpoints должны читать enrollment, а не собирать состояние из курса, assignment и frontend cache.
- Backfill обязан сверить число прохождений, завершённых уроков, попыток и итоговые статусы до переключения чтения.
