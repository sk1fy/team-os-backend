# ADR-004: Эволюция контрактов без синхронного деплоя

- Статус: принято
- Дата: 2026-07-12

## Контекст

Сервисы развёртываются независимо и во время rolling update одновременно работают две версии. Breaking change заставит обновлять потребителей синхронно и отменит главное преимущество выбранной архитектуры.

## Решение

Контракты изменяются по следующим правилам:

1. REST/OpenAPI: в `/api/v1` добавляются только новые endpoints и необязательные поля. Удаление, переименование, изменение типа или обязательности требует `/api/v2`; предыдущая версия поддерживается минимум два релиза. CI валидирует спецификацию и запускает `oasdiff breaking` относительно базовой ветки.
2. Protobuf/gRPC: номера полей никогда не переиспользуются, удалённые номера и имена помещаются в `reserved`, новые поля добавляются как совместимые optional/repeated. CI запускает `buf lint`, `buf build` и `buf breaking`.
3. События: версия входит в subject. Несовместимый payload создаёт `.v2`; издатель временно публикует обе версии, пока существуют потребители v1. Неизвестные новые поля envelope/payload игнорируются.
4. База: миграции следуют expand → migrate → contract. Сначала добавляется совместимая nullable/default колонка, затем обе версии кода могут работать, данные переносятся, и только в отдельном релизе удаляется старое поле.

Каждый endpoint, RPC и event закреплён за владельцем. Protobuf-схемы событий находятся в `contracts/events`, а издатели и потребители зафиксированы в архитектурном плане; OpenAPI является источником правды для gateway и frontend client generation.

## Последствия

- Некоторые устаревшие поля и двойная публикация временно увеличивают объём кода и данных.
- Breaking cleanup нельзя объединять с функциональным изменением в одном релизе.
- Сервис можно обновлять и откатывать без одновременного деплоя соседей.
- Исключение из правил требует отдельного ADR, плана миграции потребителей и согласованного окна несовместимости.
